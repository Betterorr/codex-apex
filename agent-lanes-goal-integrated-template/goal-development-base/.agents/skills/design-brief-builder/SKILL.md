# 设计简报构建器

目标：把产品意图、审美偏好和交互要求转成可执行的设计 Brief。

## 自然语言触发词

用户提到这些词或表达时，优先使用本 Skill：

- 设计、设计规范、设计 brief、UI、UX、交互、布局。
- 聊聊界面、讨论交互、页面怎么组织、这个界面怎么设计、用户怎么操作。
- 信息架构、页面结构、导航、面板、状态、空状态、加载态、错误态。
- 视觉方向、产品气质、界面风格、参考、反参考、截图。
- Figma 审查、设计审查、画布级审查、首屏任务、行动路径、审计信息折叠。
- 中文优先、人类可读、人话文案、程序代码信息、机器字段、技术字段外露。
- 决策卡片、研究卡片、候选卡片、复盘卡片、报告卡片、行动卡片、核心业务判断、当前判断、决策意义、机器字段卡片。
- 工具地图、一级页面、页面职责、用户路径、功能定位、入口、结果回看、验证工作台、结果工作台、内部模拟执行、沙盒执行。
- 文案语气、可访问性、响应式、密度、颜色、字体、圆角。

## 输入

优先读取：

- `docs/01-product-spec.md`
- `docs/02-design-brief.md`
- 已有原型、截图、设计稿或 UI 代码。
- `.codex/signals/`

## 核心契约

不要停留在“高级感”“简洁”“像某某产品”这种模糊描述。必须把感受翻译成布局、密度、状态、交互、视觉约束和反参考。

前端工作台类页面必须先通过用户任务和信息架构，而不是先堆字段。设计 Brief 不能只列页面字段、状态枚举和边界声明；必须说明首屏用户先看什么、下一步点哪里、完成后回到哪里、哪些是主判断、哪些只是审计证据。

所有用户可见前端必须先通过 Human-Readable Frontend Gate。设计 Brief 要明确主阅读层如何用中文表达业务含义，不能让开发把 raw enum、JSON key、内部状态名、source path、artifact path、validator status、rerun command、thread/message id、DOM/API 名、脚本名或调试字段直接放进主标题、主按钮、卡片正文、列表主列、首屏说明或导航文案。需要保留的技术信息必须设计为折叠详情、审计抽屉、开发者诊断或报告来源层。

涉及目标项目的研究/候选/复盘/报告/行动卡片时，设计 Brief 必须先定义卡片类型和决策意义，而不是从 `source_blocker_review`、`target_surface`、`execution_order`、`blocked_state`、`return anchors` 等机器字段反推 UI。每张主视图卡片必须说明：当前判断、对核心业务判断的影响、2-4 条关键证据、否决/风险条件、下一步人工动作、技术审计折叠位置。这里的“核心业务判断”只表示是否进入下一步人工判断，不得写成高风险外部执行指令、收益承诺、效果证明或生产化承诺。

涉及目标项目前端工具地图或跨页面工作台时，设计 Brief 必须先通过 Tool-Level IA Gate：从用户目标出发定义一级页面、页面职责、用户问题、卡片职责、字段如何支撑功能、入口、结果回看和状态边界。不能只描述当前已有卡片和字段，也不能把验证工作台、内部模拟执行或结果看板降格为报告里的附属字段。

涉及内部模拟执行、沙盒执行或仿真工作台时，设计 Brief 必须通过 Internal Simulation Semantics Gate：当前阶段可以是本地样本 / 静态 payload，但这不是最终产品定位；内部模拟执行不是外部真实执行、不改变真实远程资源，却应被设计成产品内部受控状态变化，会产生模拟操作、模拟结果、虚拟资源、指标曲线、规则阻断和复盘记录。真实账号、真实资源、secret、scheduler、production feed、远程写入或自动执行仍需 guardian / 用户授权。

## 项目文档中文优先

更新 `docs/*.md`、`agent-lanes/**/*.md`、泳道 worklog、dashboard、GOAL log、review/release/report 正文或模板说明时，默认写中文。专有名词、品牌、代码、命令、路径、API 名、JSON key、状态枚举、错误原文、许可证原文和外部引用标题可以保留英文或原文。

## 必须澄清的内容

- 主界面或主要操作面的信息架构。
- 页面、面板、状态、空状态、错误态、加载态。
- 用户主要操作路径。
- 首屏任务焦点、下一步行动、返回路径、主层/详情层/审计层分工。
- raw enum、source lineage、artifact path、forbidden flags、safe rerun command 等审计信息默认放在哪里折叠。
- Human-Readable Frontend Gate：哪些信息在主阅读层用中文人话表达，哪些技术字段进入折叠详情、审计抽屉、开发者诊断或报告来源层。
- 状态视觉语义：风险/阻断、复核、观察、暂缓等是否有颜色、图标、标题和动作文案差异。
- 涉及卡片时的卡片类型：`decision_card`、`evidence_card`、`workflow_card` 或 `audit_card`。
- 决策意义：当前判断、对核心业务判断的影响、关键证据、否决/风险条件、下一步人工动作和技术审计折叠。
- 工具级 IA：一级页面、每页职责、用户问题、主入口、返回路径、结果回看、状态边界和字段用途。
- 验证实验与内部模拟执行：验证工作台是否是一级工具；仿真结果工作台是否是一级内部模拟执行与结果回看入口。
- 内部模拟执行语义：本地样本阶段、产品内真实模拟操作、高风险外部执行三者如何区分。
- 正向参考和反向参考。
- 主题、密度、字体、颜色、圆角、空间层级。
- 已有原型代码是复用、参考还是废弃。

## 打回条件

遇到这些情况不要直接进入开发：

- 只有审美词，没有具体设计决策。
- 只有单一页面，没有关键状态和交互。
- 只列字段和状态枚举，没有首屏任务焦点、行动路径和返回路径。
- 把审计字段、raw enum、source lineage、artifact path 和用户主操作放在同一视觉层级。
- 主阅读层暴露 JSON key、raw enum、内部状态名、路径、命令、thread/message id、validator 字段、DOM/API 名、脚本名或调试字段。
- 用户可见文案大面积英文或中英混杂，且没有中文解释。
- 涉及研究/候选/复盘/报告/行动卡片，却没有声明卡片类型，或没有说明当前判断、核心业务判断影响、关键证据、否决/风险条件和下一步人工动作。
- 把 `source_blocker_review`、`target_surface`、`execution_order`、`blocked_state`、`return anchors` 等机器字段作为主标题、主按钮或主阅读结构。
- 质量 blocker 只作为字段存在，没有转译成用户能理解的“暂不支持下一步判断 / 继续观察 / 进入证据复核 / 排除”等流程状态。
- 工具地图只讲已有卡片和字段，没有从用户目标定义一级页面、页面职责、用户路径、功能定位、入口和结果回看。
- 把当前阶段的本地样本 / 静态 payload 边界写成最终产品只读、不建议、不执行的永久定位。
- 把内部模拟执行等同外部真实执行，或反过来把内部模拟执行降格成只读审计文本，没有模拟操作、模拟结果、虚拟资源、指标曲线和复盘记录。
- 没有说明桌面和窄屏分别如何控制信息密度。
- 原型或截图存在，但没有说明复用边界。
- 设计要求和产品范围冲突。
- UI 复杂度超出第一版范围但没有取舍。

## 输出

更新 `docs/02-design-brief.md`，至少包含：

- 产品气质。
- 参考与反参考。
- 信息架构。
- 关键页面或功能面。
- 交互规则。
- 视觉约束。
- 可访问性和响应式要求。
- Figma/设计审查口径：首屏任务、行动路径、主次分层、审计折叠、状态语义、人话文案、窄屏可用和风险提示。
- Human-Readable Frontend Gate：主阅读层中文优先、技术字段转译成人话、审计/调试/路径/命令/ID 信息折叠。
- Decision-Usefulness Gate：涉及卡片时，声明卡片类型，并写清当前判断、核心业务判断影响、关键证据、否决/风险条件、下一步人工动作和技术审计折叠规则。
- Tool-Level IA Gate：涉及工具地图或跨页面工作台时，写清一级页面结构、页面职责、用户路径、入口、结果回看、状态边界和字段支撑功能。
- Internal Simulation Semantics Gate：涉及内部模拟执行或沙盒执行时，写清本地样本阶段、产品内受控模拟操作和高风险外部执行的边界。

如果设计改变了产品行为，同步更新：

- `docs/01-product-spec.md`
- `docs/CHANGELOG.md`

如果本轮作为 design 泳道 completion callback 回报，JSON 必须额外包含：

- `active_user_loop`: 本轮设计服务的用户闭环。
- `loop_impact`: `advanced`、`blocked`、`regression_fixed` 或 `neutral`；普通设计任务不应只停在 `neutral`。
- `blocking_concerns`: 真正阻塞当前用户闭环的设计问题。
- `backlog_concerns`: 不阻塞当前闭环、应进入 backlog 或阶段合并审查的设计质量项。
- `recommended_next_type`: `vertical_loop`、`blocker_fix`、`boundary_review`、`backlog_only` 或 `defer_review_until_boundary`。

## 信号记录

当用户纠正审美方向、信息架构、原型复用方式或交互边界时，在 `.codex/signals/` 记录 signal。

## 完成门禁

只有当 `prototype-builder` 或 `dev-planner` 可以不发明主布局、状态、交互模型和视觉方向时，设计 Brief 才算完成。

## 与前端工作流 Skill 的协作

当设计任务涉及前端底座、复杂工作台、图表/事件/表格/流程组件、数据层级、UI 组件库、目录结构、模块边界、Design Token、主题、多语言或规划端/设计端共用前端流程时，先使用 `frontend-workflow-planner`。

设计 Brief 必须吸收它输出的 `design_language`、`data_contract`、`module_boundaries`、`reuse_rules`、`design_tokens` 和 `acceptance_evidence`，再写入 `docs/02-design-brief.md`。如果这些字段缺失，设计泳道应打回澄清，不要直接进入页面设计。
