# 原型构建器

目标：把需求和设计 Brief 转成可供开发复用或验收的原型资产。

## 项目文档中文优先

更新 `docs/*.md`、`agent-lanes/**/*.md`、泳道 worklog、dashboard、GOAL log、review/release/report 正文或模板说明时，默认写中文。专有名词、品牌、代码、命令、路径、API 名、JSON key、状态枚举、错误原文和外部引用标题可以保留英文或原文。

## 自然语言触发词

用户提到这些词或表达时，优先使用本 Skill：

- 原型、设计稿、高保真、线框图、可点击原型、页面清单。
- Figma、设计工具导出、截图转实现、原型代码、静态原型。
- 复用原型、不要重写设计、按这个原型开发、原型验收。
- 设计代码能不能直接用、哪些页面要覆盖、状态覆盖、交互演示、demo 页面。

## 输入

优先读取：

- `docs/01-product-spec.md`
- `docs/02-design-brief.md`
- 已有原型、设计代码、截图、Figma/设计工具导出物。
- `docs/00-project-state.md`

## 核心契约

原型不是“灵感参考”就结束。必须明确它在开发中的地位：直接复用、部分复用、仅作视觉参考，还是废弃重做。

## 必须澄清的内容

- 原型覆盖哪些页面、状态和主流程。
- 原型是否需要可点击交互。
- 原型代码是否作为生产代码起点。
- 哪些设计细节必须还原，哪些可由开发阶段调整。
- 原型中额外生成的功能是否进入产品范围。

## 打回条件

- 原型缺少主流程关键状态。
- 原型和产品需求明显不一致。
- 原型代码存在但没有复用策略。
- 设计工具擅自加的功能没有被确认。
- 不能说明原型如何被开发计划消费。

## 输出

根据项目状态更新：

- `docs/02-design-brief.md`：补充原型约束、页面状态和复用边界。
- `docs/03-dev-plan.md`：说明原型如何进入开发计划。
- `docs/00-project-state.md`：记录原型资产位置和可信度。

## 信号记录

当用户纠正“原型只是参考还是要复用”、设计工具加功能、页面覆盖范围时，在 `.codex/signals/` 记录 signal。

## 完成门禁

只有当开发计划能明确使用原型资产，并且不会误把高保真代码当成可随意丢弃的参考时，原型环节才算完成。
