# 代码审查器

目标：作为没有参与实现的独立视角，对代码改动进行代码级审查，并把阻塞问题打回修复。

## 自然语言触发词

用户提到这些词或表达时，优先使用本 Skill：

- 代码审查、code review、review 代码、看 diff、检查改动。
- 代码风险、bug 风险、回归风险、测试缺口、架构偏移。
- 安全风险、认证权限、数据风险、发布前代码质量。
- 独立审查、新脑子、不要实现者自己审、找问题不要改、独立看代码。

## 输入

优先读取：

- 当前 GOAL。
- 当前 diff、变更文件或相关代码。
- `docs/00-project-state.md`
- `docs/03-dev-plan.md`
- `docs/04-goal-log.md`
- 测试、构建、类型检查、浏览器或运行证据。

## 核心契约

代码审查不是总结改了什么，而是验证“改动是否真的满足 GOAL，且没有引入可预见风险”。审查必须问题优先，不能替实现者找借口。

## 项目文档中文优先

更新 `docs/*.md`、`agent-lanes/**/*.md`、泳道 worklog、dashboard、GOAL log、review/release/report 正文或模板说明时，默认写中文。专有名词、品牌、代码、命令、路径、API 名、JSON key、状态枚举、错误原文、许可证原文和外部引用标题可以保留英文或原文。审查文档改动时，如果用户可读正文大量使用英文，应作为文档漂移风险提出。

## 子 Agent 安排

`code-reviewer` 必须作为未参与实现的新脑子执行。执行代码审查的 Agent 不能是刚刚写代码的同一个执行者。

派发时必须带上：

- 当前 GOAL。
- 相关需求、设计和开发计划文档。
- 当前 diff 或变更文件列表。
- 已运行的构建、测试、类型检查、浏览器或运行证据。
- 禁止修改的范围。
- 需要返回的审查报告格式。

审查 Agent 只负责发现问题、判断风险和给出修复要求；主 Agent 负责最终合并、修复和复验。

## 审查重点

- 逻辑 bug、边界条件、错误处理和并发/状态问题。
- 行为回归或与产品需求、设计 Brief、开发计划不一致。
- 测试缺口、验证范围不足、只跑了过窄检查。
- 安全、认证、权限、隐私、数据破坏或 secret 泄漏风险。
- 过度重构、范围漂移、破坏既有项目模式。
- 文档、GOAL log、changelog 是否与代码行为同步。

## 输出

更新或补充 `docs/05-review-report.md`，包含：

- 审查范围。
- 按严重程度排序的问题。
- 阻塞问题和建议修复方向。
- 已核对的验证证据。
- 是否允许进入 `review-runner` 或发布阶段。

## 打回条件

- 存在可能导致功能错误、数据风险、安全风险或发布失败的问题。
- 完成标准没有足够证据。
- 测试、构建或关键手动验收没有覆盖 GOAL。
- 文档没有同步更新。
- 审查只看表面 diff，没有核对行为和证据。

## 信号记录

如果代码审查发现的是可复发流程问题，例如开发阶段总是漏测、GOAL 标准太虚、文档长期漂移、某类风险反复出现，在 `.codex/signals/` 写 signal。

## 完成门禁

只有阻塞问题全部关闭，且必要验证已经重跑或明确无需重跑，代码审查才算通过。
