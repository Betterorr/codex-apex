# GOAL 方法论抽象版

## 1. 核心定义

GOAL 方法论是一套让 AI Agent 自主推进项目的项目操作系统。

它不是把每一步写死给 AI 执行，而是由人定义目标、完成标准和验收方式，再让 AI 自己规划、执行、纠错、复审和迭代。

一句话概括：

> 人负责决策和验收，AI 负责规划、执行、修正和沉淀。

核心公式：

```text
清晰目标 + 可验证标准 + 文档上下文 + 自动审查门禁 + 失败信号进化 = 可持续自主开发循环
```

## 2. 范式变化

传统 AI 编程更像“指令式协作”：

```text
人拆步骤 -> 人写提示词 -> AI 执行局部任务 -> 人检查 -> 人继续指挥
```

GOAL 方法论更像“目标式委托”：

```text
人定义目标和验收标准 -> AI 自主拆解 -> AI 执行 -> AI 自查/他查 -> 不通过则自动修正 -> 通过后交付证据
```

关键变化：

- 从重流程转向轻流程。
- 从规定“怎么做”转向规定“什么算做好”。
- 从人盯过程转向系统卡验收。
- 从长上下文聊天转向文档驱动。
- 从预设大量规则转向失败后沉淀规则。

## 3. 人与 AI 的分工

人的职责只有两类：

1. 做决定：选择方向、边界、取舍、优先级。
2. 做验收：判断交付结果是否真的满足目标。

AI 的职责包括：

1. 追问需求。
2. 生成文档。
3. 规划开发阶段。
4. 编写代码。
5. 调用工具。
6. 自测和修复。
7. 派发独立审查。
8. 更新项目文档。
9. 记录失败信号。
10. 抽象规则改进建议。

## 4. 总体流程

完整流程可以抽象为六个主环节：

```text
需求澄清 -> 设计规范 -> 原型/设计图 -> 开发计划 -> 开发执行 -> 发布交付
```

每个环节都有一个专门 Skill，产出一份可传递给下游的文档。

| 环节 | 目标 | 产物 | 下游用途 |
| --- | --- | --- | --- |
| 需求澄清 | 把模糊想法逼问成可开发需求 | 产品需求文档 | 输入给设计和开发计划 |
| 设计规范 | 把主观感受翻译成设计决策 | 设计 Brief | 输入给原型和前端实现 |
| 原型设计 | 生成可视化高保真方案 | 原型代码/设计稿 | 输入给开发执行 |
| 开发计划 | 把项目拆成可验收阶段 | Dev Plan | 输入给开发 Agent |
| 开发执行 | 按计划实现功能 | 代码、测试、变更记录 | 输入给审查和发布 |
| 发布交付 | 做隐私、安全、打包、部署检查 | Release Report | 最终交付和回滚依据 |

## 5. 文档驱动机制

这套方法论的上下文不是主要靠聊天记录，而是靠文档。

每一环节产出一份稳定文档，下一个环节读取它继续工作：

```text
Product Spec -> Design Brief -> Prototype -> Dev Plan -> Code -> Release Notes
```

文档的作用：

- 固化跨 session 上下文。
- 避免模型因为长对话丢失重点。
- 让新 session 可以低成本接续。
- 让每次迭代都有可追溯记录。
- 让项目不依赖某一次对话状态。

一个重要原则：

> 每次功能迭代后，相关文档必须同步更新。

即使只是一个交互优化，也可能需要同步更新需求、设计、开发计划和变更记录。这样后续 Agent 才不会基于过期信息继续开发。

## 6. Skill 设计原则

Skill 不是详细操作手册，而是任务契约。

一个好的 Skill 主要写三类内容：

1. 什么时候使用。
2. 什么结果算合格。
3. 哪些红线不能碰。

不应该写太多：

- 第一步做什么。
- 第二步做什么。
- 每个细节怎么操作。
- 所有可能情况的硬编码流程。

更好的写法：

```text
少写“怎么做”，多写“什么算好”。
少写流程细节，多写验收标准。
少写预判规则，多写真实失败后的规则。
```

但这不等于 Skill 要很薄。相反，关键判断标准可以写得很厚，例如：

- 什么需求回答可以接受。
- 什么回答必须追问。
- 什么设计描述太模糊。
- 什么边界不能遗漏。
- 什么完成状态不能放行。

## 7. Agent 编排原则

默认由一个主 Agent 贯穿项目，不要把任务过早拆给大量子 Agent。

只有两种情况适合派发子 Agent：

1. 需要一颗干净的新脑子，例如代码审查、产品审查、安全审计。
2. 多块任务互不依赖，可以并行推进。

子 Agent 的定位：

```text
子 Agent 是另一颗脑子，不是另一个文件夹。
```

主 Agent 必须保留最终合并和拍板权。

派发时必须带走完整上下文：

- 当前目标。
- 相关文档。
- 变更范围。
- 验收标准。
- 禁止事项。
- 需要返回的证据格式。

## 8. GOAL 机制

GOAL 是驱动自主循环的最小控制单元。

一条合格的 GOAL 必须包含：

1. 目标：要完成什么。
2. 完成标准：做到什么才算完成。
3. 验收方式：用什么证据证明完成。
4. 失败处理：如果没通过要如何继续修。

基础模板：

```text
GOAL:
完成 [具体目标]。

完成标准:
- [标准 1，可验证]
- [标准 2，可验证]
- [标准 3，可验证]

验收方式:
- 提供 [测试/构建/截图/日志/审查报告]。
- 说明修改了哪些文件。
- 说明还有哪些限制或风险。

循环规则:
- 如果任一标准未通过，继续修复。
- 修复后重新验证。
- 直到全部标准通过，才能报告完成。
```

GOAL 的关键不是“喊口号”，而是把结束条件写清楚。

坏 GOAL：

```text
帮我把项目做好。
```

好 GOAL：

```text
完成 Phase 1 的用户登录和受保护路由。
完成标准是：构建通过、登录/登出可用、未登录访问受保护页面会重定向、相关 API 有鉴权、E2E 测试通过。
验收方式是：贴出构建结果、测试结果、关键页面截图和代码审查结论。
```

## 9. 开发闭环

每个开发任务都必须形成闭环：

```text
开发 -> 自测 -> 独立审查 -> 修复 -> 复审 -> 文档更新 -> 完成
```

其中最重要的是独立审查。

审查 Agent 必须是新派发的，不能参与原始开发。原因是：

- 没有“代码是我写的”的偏见。
- 更容易发现遗漏。
- 更适合作为门禁。

审查不通过时，不能口头解释后放行，必须修到通过。

## 10. 自动门禁

能用程序判断的规则，不要只写在提示词里。

例如：

- 构建失败不能完成。
- 测试失败不能完成。
- 审查失败不能完成。
- 必需文件不存在不能进入下一阶段。
- 发布前隐私审计未完成不能发布。

这些应该做成自动门禁，而不是靠 Agent 自觉。

抽象原则：

```text
规则负责表达标准，程序负责卡住底线。
```

## 11. 自进化机制

自进化不是不断往规则里堆内容，而是把真实失败转成规则优化。

流程：

```text
用户纠正/失败发生 -> 记录 signal -> 新 session 消化 signal -> 抽象规则建议 -> 用户确认 -> 写入 Skill/规则/问题库 -> 清空 signal
```

一条规则值得留下，必须满足：

1. 来自真实失败或真实纠正。
2. 删除后问题大概率复发。
3. 能提升后续同类任务表现。
4. 不会明显限制模型能力。

规则也要会删除：

- 长期不用的规则。
- 过时的规则。
- 与新流程冲突的规则。
- 只是在重复常识的规则。

核心原则：

> 规则是从失败里长出来的，不是坐着预判出来的。

## 12. 轻流程与厚标准

这套方法论最重要的结构是：

```text
轻流程 + 厚标准
```

轻流程：

- 不写过细操作步骤。
- 不把 Agent 拆成复杂流水线。
- 不提前规定所有实现路径。
- 不限制模型自主规划能力。

厚标准：

- 明确什么算需求清楚。
- 明确什么算设计可执行。
- 明确什么算一个 Phase 可验收。
- 明确什么算开发完成。
- 明确什么情况必须打回。
- 明确交付时必须给出哪些证据。

## 13. 项目节奏

推荐节奏：

```text
一个 session 只做一个清晰目标。
一个 GOAL 驱动一个闭环。
一个闭环完成后更新文档。
下一个功能开新 session。
```

这样做的好处：

- 上下文小。
- 注意力集中。
- 交付边界清晰。
- 更容易回滚和审计。
- 后续 Agent 可以从文档重新接续。

## 14. 可复用系统架构

可以把这套方法论实现成如下结构：

```text
project/
  AGENTS.md
  docs/
    product-spec.md
    design-brief.md
    dev-plan.md
    release-report.md
    changelog.md
  .agents/
    skills/
      product-spec-builder/
      design-brief-builder/
      prototype-builder/
      dev-planner/
      dev-builder/
      release-builder/
      goal-creator/
      bug-fixer/
      code-reviewer/
      evolution-runner/
  .codex/
    subagents/
    hooks/
    signals/
    gates/
```

最小可用版本不一定需要全部模块，但至少要有：

- 顶层项目规则。
- 需求文档。
- 设计/实现约束文档。
- 开发计划。
- GOAL 模板。
- 审查门禁。
- 变更记录。

## 15. 方法论的本质

GOAL 方法论的本质不是提示词技巧，而是流程设计。

它真正要解决的问题是：

- 如何让 AI 不只是写代码，而是推进项目。
- 如何让 AI 跨 session 不丢上下文。
- 如何让 AI 自己发现问题并修复。
- 如何让人的参与从“盯过程”变成“做决策”。
- 如何让系统越用越懂你，但规则不无限膨胀。

最终范式：

```text
人定目标，AI 跑循环。
人做验收，系统做门禁。
文档传上下文，失败长规则。
```
