# 产品需求构建器

目标：把模糊想法逼问成可开发、可验收、可传递给设计和开发计划的需求文档。

## 自然语言触发词

用户提到这些词或表达时，优先使用本 Skill：

- 需求、产品需求、PRD、需求文档、产品文档、规格说明。
- 聊聊需求、探讨需求、讨论需求、梳理需求、需求脑暴、产品脑暴。
- 我有个想法、我想做一个、把想法变成需求、帮我整理需求、先问清楚、需求澄清。
- 范围、边界、做与不做、第一版、MVP、用户故事、这个产品怎么定义。
- 这个功能要不要做、需求取舍、方案取舍、第一版先做什么、先别开发、先把需求想清楚。
- 验收标准、功能列表、功能清单、产品流程、业务流程、目标用户、核心场景、用户痛点、主流程、非目标。

## 输入

优先读取：

- `docs/00-project-state.md`
- `docs/01-product-spec.md`
- `docs/02-design-brief.md`
- `.codex/signals/`

## 核心契约

少写“怎么实现”，多问“什么算做成”。需求文档必须让下游 Agent 不靠猜测就能继续做设计和开发计划。

## 项目文档中文优先

更新 `docs/*.md`、`agent-lanes/**/*.md`、泳道 worklog、dashboard、GOAL log、review/release/report 正文或模板说明时，默认写中文。专有名词、品牌、代码、命令、路径、API 名、JSON key、状态枚举、错误原文、许可证原文和外部引用标题可以保留英文或原文。

## 必须澄清的内容

- 用户是谁，主要任务是什么。
- 主流程从开始到成功结束是什么。
- 第一版明确做什么、不做什么。
- 什么输出、状态或行为能证明成功。
- 哪些依赖、权限、账号、数据、模型、平台或人工步骤会影响交付。
- 哪些约束由用户决定，哪些可以交给 Agent 自主规划。

## 打回条件

遇到这些情况不要直接写成需求，必须追问或标成未决问题：

- 只有口号，没有具体用户动作。
- 只有功能名，没有验收标准。
- “都支持”“越快越好”“高级感”这类答案会扩大范围但没有边界。
- 外部依赖不清楚，例如账号、API、数据来源、部署环境。
- 用户把可配置策略和产品固定需求混在一起。

## 输出

更新 `docs/01-product-spec.md`，至少包含：

- 问题定义。
- 目标用户。
- 主流程。
- 做与不做。
- 需求列表和验收标准。
- 外部依赖。
- 未决问题。

同步更新 `docs/00-project-state.md` 的当前阶段、已验证事实和下一个建议 GOAL。

## 信号记录

当用户纠正了追问方向、产品边界、可配置能力或问题库判断时，在 `.codex/signals/` 写一条简短 signal，供 `evolution-runner` 后续消化。

## 完成门禁

只有同时满足以下条件，才算完成：

- 需求不是聊天摘要，而是可开发文档。
- 每个核心需求都有验收方式。
- 范围和非范围都明确。
- 未决问题被显式列出，而不是被隐藏。
- 下游可以进入 `design-brief-builder` 或 `dev-planner`。
