# 自进化执行器

## Agent Lanes V2 瘦身约束

机制进化优先减少规则、镜像页面和传输噪音；默认一次只处理一个机制主题。新增规则前先确认现有 Product Value Gate 不能表达该约束，避免继续叠加同类 shell。项目本地验证通过后再同步可移植模板。

目标：根据真实失败、用户纠正和当前项目的稳定需求变化，主动净化并进化本地框架，让其他 Skill 更适应本项目，同时避免每次“继续”都打断主流程、避免规则膨胀和越权改造。

## 自然语言触发词

用户提到这些词或表达时，优先使用本 Skill：

- 进化、自进化、自净化、规则更新、沉淀规则。
- signals、失败信号、复盘规则、把纠正变成规则。
- 删规则、规则膨胀、问题库、调优框架。
- 新 session 消化信号、规则建议、这条以后记住。
- 优化其他 Skill、自动进化本地 Skills、让 Skill 更适应项目。
- 净化规则、净化 schema、调整 Skill 边界。
- 边界写死、授权后还被挡住、不要把边界写成禁止、真实接入被机制拦住、动态授权闸门。

## 输入

读取：

- `.codex/signals/`
- `AGENTS.md`
- `.codex/hooks/skill-hooks.md`
- `.codex/gates/skill-mechanism-check.ps1`
- `.agents/skills/` 中相关本地 Skill。
- 与 signal 或需求变化有关的项目文档。

## 核心契约

自进化不是堆规则。它净化的是本项目的工作机制，包括：

## 项目文档中文优先

更新 `docs/*.md`、`agent-lanes/**/*.md`、泳道 worklog、dashboard、GOAL log、review/release/report 正文或模板说明时，默认写中文。专有名词、品牌、代码、命令、路径、API 名、JSON key、状态枚举、错误原文、许可证原文和外部引用标题可以保留英文或原文。邮局合并信、outbox、dashboard 这类人类可读投递产物也必须保证 Windows/PowerShell/桌面预览中文可读。进化 Skill、hook 或门禁时，必须同步维护这条语言规则，不能只改聊天回复风格。

- 触发词和路由：让自然语言更准确地命中正确 Skill。
- Skill 边界：减少重叠、误触发、缺步骤和职责漂移。
- Skill 契约：补齐输入、输出、完成门禁、打回条件和证据要求。
- 项目适配：把当前项目的真实需求、技术栈、验证方式、发布方式写进相关 Skill。
- Skill 本地化：维护项目机制运作的 Skill 必须创建或改造到 `.agents/skills/<skill-name>/`；公共 Skill 目录只能作为方法来源或验证器来源，不能作为项目机制 Skill 的最终安装位置。
- 机制门禁：把可检查的 Skill 结构要求写进 `.codex/gates/skill-mechanism-check.ps1`。
- 过时规则：删除或收紧已经不适用、重复常识、阻碍自主推进的规则。

自进化不是净化业务代码、产品方向或用户偏好。业务功能、架构选型、外部服务、凭据、部署和数据迁移必须由对应开发、发布或审查 Skill 处理。

## 触发边界

本 Skill 不在每次“继续/下一步”时自动运行。只有下面情况才运行：

- 显式触发：用户直接要求进化、净化、调整 Skill、优化 hook、沉淀规则、处理 signals。
- 用户纠正了流程，并且这条纠正会影响后续同类任务。
- 同一类误路由、漏步骤、漏门禁、漏文档更新、错误完成声明重复出现，或已经阻塞当前 GOAL。
- 项目文档出现稳定变化，例如技术栈、运行命令、验收方式、发布方式，而相关 Skill 尚未同步。
- `.codex/signals/` 有非 `.gitkeep` 的未处理 signal，且内容指向本地 Skill、hook、门禁或流程规则。
- 某个 Skill 明显不适合当前项目阶段，导致 `project-orchestrator` 无法可靠选择下一步。

不要运行本 Skill：

- 用户只是说“继续/下一步”，且当前应推进需求、设计、计划、开发、验证或审查。
- 问题只是一次性业务判断、产品方向讨论、审美偏好或功能取舍。
- 当前失败能由 `bug-fixer` 修复，且不暴露可复用流程缺口。
- 当前缺口能由 `product-spec-builder`、`dev-planner`、`gate-runner` 或 `review-runner` 正常完成。
- 没有证据说明某个规则会复发。

## 自动落地权限

满足以下条件时，可以不等用户确认，直接编辑本地规则、hook、门禁或 Skill：

- 改动只发生在 `AGENTS.md`、`FRAMEWORK.md`、`.codex/`、`.agents/skills/`、`docs/` 中的流程文档。
- 改动来自真实用户纠正、失败 signal、项目文档更新或已验证的流程缺口。
- 改动能让 Skill 更适应当前项目，而不是加入泛泛常识。
- 改动不会扩大外部权限、不会引入新依赖、不会触碰密钥、不会修改业务代码。
- 改动范围小，可解释，可通过本地校验脚本验证。
- 改动只影响一个清晰问题，不顺手重写无关 Skill。

必须先停下来请求用户确认的情况：

- 要删除整个 Skill 或大幅改写多个 Skill 的职责边界。
- 要改变“人负责方向决策和最终验收”的总原则。
- 要修改业务代码、依赖、外部 API、认证、部署、数据或凭据。
- 要新增会自动执行外部网络、发布、支付、删除文件或改变远程状态的机制。
- 信号来源不清楚，或可能只是一次性偏好而不是可复用规则。
- 要把用户的一次性产品取舍写成长期流程规则。

## 处理流程

1. 读取 signals、项目状态、hook 路由和相关 Skill。
2. 判断触发原因属于：显式触发、真实失败、用户纠正、稳定需求变化、重复摩擦、过时规则、一次性事件。
3. 选择净化对象：触发词、Skill 边界、输入输出、完成门禁、打回条件、机制门禁、项目文档。
4. 合并重复 signal，删除不值得固化的一次性事件。
5. 先做最小有效改动。在自动落地权限内直接修改文件；越界时只输出变更方案和需要用户确认的问题。
6. 运行 `.\scripts\validate-skills.ps1`、`.\scripts\check-framework.ps1`、`.\scripts\check-skill-mechanism.ps1`。
7. 更新 `docs/00-project-state.md` 或相关文档，记录自进化结果和未决风险。
8. 清空、归档或改写已处理 signal，避免重复处理。

## 保留规则的标准

只有满足这些条件才保留为规则：

- 来自真实失败、用户纠正或已经写入项目文档的稳定需求变化。
- 删除后问题很可能复发。
- 能提升后续同类任务表现。
- 有明确适用边界，知道什么时候不该用。
- 能落到某个 Skill、hook、门禁或文档，不只是口号。
- 不会明显限制 Codex 自主规划能力。
- 对“什么时候不用”也有清晰边界。

## 删除规则的标准

主动删除或收紧：

- 长期不用的规则。
- 与当前项目新流程冲突的规则。
- 重复常识的规则。
- 让 Agent 过度保守、总是追问或不敢执行的规则。
- 只适用于旧项目模板、不适用于当前项目的规则。
- 让多个 Skill 职责重叠、互相抢触发的规则。
- 把一次性业务偏好错误固化成长期流程的规则。

## 进化其他 Skill 的标准

可以进化其他 Skill，但每次必须满足“改动必要性”并说明：

- 被进化的 Skill 是哪个。
- 触发这次进化的证据是什么。
- 修改的是触发、边界、流程、输出、门禁还是打回条件。
- 为什么这会让它更适应当前项目。
- 为什么不由当前业务 Skill 直接处理。
- 哪些内容没有改，避免越界。

改 Skill 时优先做小补丁：

- 补触发词，而不是重写整个 Skill。
- 补项目约束，而不是加入泛用方法论。
- 补完成门禁和证据要求，而不是增加空泛原则。
- 从公共 Skill 或外部 Skill 吸收方法时，先改造成项目本地版，产物必须回写本项目既有文档/泳道体系。
- 删除或合并重复段落，而不是叠加更多规则。

不适当的 Skill 修改包括：

- 因一次聊天偏好重写长期流程。
- 为了覆盖所有可能表达而无限堆触发词，导致多个 Skill 抢触发。
- 把业务实现细节写进通用执行 Skill，导致后续阶段被旧决策绑死。
- 把本该写进 `docs/01-product-spec.md`、`docs/02-design-brief.md` 或 `docs/03-dev-plan.md` 的项目内容写进 Skill。
- 同时修改多个无关 Skill，无法解释每个改动的证据。

## 与项目总调度器的关系

`project-orchestrator` 在“下一步/继续”时只检查是否满足自进化触发边界。存在 `.codex/signals/` 不等于必须运行本 Skill；signal 必须具体指向本地 Skill、hook、门禁或流程规则，或者已经阻塞当前 GOAL。

没有具体 signal、没有用户纠正、没有重复摩擦、没有稳定项目约束变化时，不要为了形式自进化。若只是发现小问题但不阻塞当前 GOAL，先完成当前主任务，再记录 signal 或在输出里建议后续处理。

## 输出

输出必须包含：

- 自我纠正触发原因。
- 净化对象和边界。
- 已自动落地的文件修改，或需要用户确认的越界修改。
- 被进化的 Skill 及原因。
- 验证命令和结果。
- 已处理 signal 的去向。

如果本轮作为 evolution 泳道 completion callback 回报，JSON 必须额外包含：

- `active_user_loop`: 本轮机制改动服务或保护的用户闭环。
- `loop_impact`: `advanced`、`blocked`、`regression_fixed` 或 `neutral`；机制优化通常应说明它解除的是节奏、可读性、回报或调度阻塞。
- `blocking_concerns`: 真正阻塞当前用户闭环或调度闭环的机制问题。
- `backlog_concerns`: 不阻塞当前闭环、应进入后续模板/门禁改进的事项。
- `recommended_next_type`: `vertical_loop`、`blocker_fix`、`boundary_review`、`backlog_only` 或 `defer_review_until_boundary`。

## 完成门禁

完成前必须满足：

- 每条已处理 signal 都被转换成落地改动、明确拒绝或保留待确认。
- 所有直接修改都在自动落地权限内。
- 已运行 Skill 校验、框架校验和机制门禁，或说明无法运行的原因。
- 相关 hook、Skill description 和正文触发词保持一致。
- 没有把一次性事件膨胀成长期规则。
