# 前端质量执行器

目标：吸收适合目标项目的前端设计方法：`interface-design`、`frontend-design` 和 Figma 设计审查视角。`interface-design` 负责工具型界面的信息架构、密度、导航、状态和设计系统；`frontend-design` 只吸收“避免通用 AI 模板感、让视觉选择来自产品语境”的 craft 方法；Figma 视角用于强制检查用户阅读优先级、主线行动路径、组件层级、状态语义、审计信息折叠和窄屏可用性。所有规则必须服从目标项目的产品定位和 GOAL 文档流。

## 吸收边界

- 默认吸收 `interface-design`：复杂工具、dashboard、数据产品、运营后台和交互工作台不是营销页，必须安静、可扫描、可比较、适合反复操作。
- 有条件吸收 `frontend-design`：只吸收产品语境、视觉差异化、排版/层级/状态 craft；不吸收 landing page、装饰性 hero、过度动效、泛紫色渐变或为“惊艳”牺牲研究效率的倾向。
- 强制吸收 Figma 设计审查视角：当存在用户可见页面、截图、Figma 文件、复杂工作台、主线验收或用户反馈“看不懂/太乱/像日志/没有主次”时，必须使用 Figma 式审查清单。即使没有 Figma 文件，也要按画布审查口径检查当前截图和本地页面。
- 强制执行 Human-Readable Frontend Gate：所有用户可见前端默认中文优先，主阅读层必须用人能理解的业务语言解释状态、判断、证据、风险和下一步。raw enum、JSON key、内部状态名、source path、artifact path、validator status、rerun command、thread/message id、DOM/API 名、脚本名、调试字段和 provider/debug 细节不得直接出现在主标题、主按钮、卡片正文、列表主列、首屏说明或导航文案里；这些只能进入折叠详情、审计抽屉、开发者诊断或报告来源层。
- 不默认吸收 `frontend-ui-ux-engineer`：它有可用的可访问性、状态和 polish 清单，但与本 Skill 已有规则重叠较高，且示例偏通用组件/电商卡片，暂不进入默认机制。
- 不默认吸收 `app-ui-design`：它偏 iOS/Android 移动端平台指南；目标项目若主要是 Web 工作台，需要时再临时参考。

## 核心契约

- 项目文档中文优先：设计说明、验收问题、截图结论和建议动作写中文；路径、命令、DOM/API 名和错误原文保留原文。
- 前端质量结论必须回写目标项目既有文档、artifacts 或泳道 workspace，不产出孤立设计稿体系。
- 如果发现 dashboard、UI 验收或截图规则存在可复用缺口，记录为 signal 或交给进化泳道，不在本 Skill 内顺手扩大机制改造。

## Figma 设计审查硬门槛

前端设计和验收不能只检查字段是否齐全、边界是否声明、截图是否存在。只要页面承载用户主线任务，就必须按 Figma/专业产品设计审查视角回答这些问题：

- 首屏任务焦点：用户打开页面后，是否能在 5 秒内知道“我现在该先看什么、先点哪里、为什么”。
- 行动路径成线：页面是否把目标项目的 `入口 -> 对象详情 -> 证据/分析 -> 行动/复盘 -> 报告/结果回看` 主线串起来，而不是把按钮散落在卡片里。
- 主次分层：主阅读层只放当前判断、最高优先级 blocker、下一步动作和必要边界；raw enum、source_lineage、return_anchors、artifact path、safe rerun command、forbidden flags 默认进入详情抽屉、Info Popover 或 Report source。
- 状态语义可见：source blocker、rule review、observation、pause candidate 等状态必须在颜色、图标、标题、动作文案和卡片结构上形成明显差异，不能 12 张卡同权重平铺。
- 人话文案：主层不得直接暴露 `source_blocker_review`、`blocked_by_source_quality`、`target_surface` 这类机器字段；必须翻成用户能判断动作的中文，例如“数据源受限，先不要进入下一步”“去标的详情复查规则证据”。
- 中文优先：用户可见 UI 文案、空错态、状态说明、风险提示和下一步动作默认中文；英文只保留品牌、代码、API 名、必要状态枚举或开发者详情。英文技术词若必须保留，主层要配中文解释。
- 窄屏可用：窄屏首屏只显示当前分组摘要、当前 1-2 个待处理项、最高风险和下一步按钮；审计详情必须折叠，不能把桌面审计报告纵向堆叠。
- 风险提示独立：`513100`、source limitation、quality blocker、not reviewed、not production-ready 这类信息必须成为可见风险提示，而不是淹没在卡片字段里。
- 颜色承担语义：风险/阻断、复核、观察、暂缓必须有稳定语义色和非颜色辅助标识；浅色装饰边框不能算状态设计。

如果以上任一 P1 问题存在，前端质量结论不能写成“UI 通过”或“用户可用”。应标记为 `DONE_WITH_CONCERNS` 或 `BLOCKED`，并把问题回写到 `docs/02-design-brief.md`、`docs/05-review-report.md` 或泳道 workspace。

## Decision-Usefulness Gate / 决策意义门槛

涉及目标项目的研究卡片、候选卡片、复盘卡片、报告卡片、行动卡片或类似“帮助用户判断下一步”的卡片时，前端质量不能只看字段齐全、validator 通过、anchor 可见和边界可追溯。主阅读层必须先回答这些问题，否则不能标为前端可用：

- 当前判断：这张卡给用户的当前结论是什么，例如“暂不支持核心业务判断”“继续观察”“进入证据复核”“排除”。这不是高风险外部执行指令，而是业务流程状态。
- 对核心业务判断的影响：它为什么影响是否进入下一步判断，是阻断、支持、削弱，还是只提供背景。
- 关键证据：最多 2-4 条用户能读懂的证据，不允许只展示 raw enum、JSON key、source path 或内部状态名。
- 否决/风险条件：哪些信息会让用户停止继续判断，例如 source limitation、data_quality_blocker、样本质量未审、证据冲突或风险过高。
- 下一步人工动作：用户下一步点哪里、看什么、回哪里，而不是机器字段里的 `target_surface`。
- 技术审计折叠：source lineage、return anchors、forbidden capabilities、raw enum、validator status、rerun command 必须进入折叠层、详情层或报告来源层。

卡片类型必须先声明：

- `decision_card`：帮助用户理解是否进入下一步业务判断、继续观察或排除。
- `evidence_card`：展示证据和质量限制。
- `workflow_card`：说明下一步流程，但必须标明“这不是决策结论”。
- `audit_card`：只给技术追溯，不应出现在主阅读层。

如果卡片不是 `decision_card`，主视图必须明确写出“本卡不支持核心业务判断，只用于证据复核 / 流程导航 / 技术审计”。如果用户看完仍不知道“现在能否进入下一步判断”，或质量 blocker 只是字段没有转译成用户语义，应标为 P1/P2 体验阻塞，而不是 backlog polish。

## Tool-Level IA Gate / 工具级信息架构门槛

前端质量审查不能只审卡片、字段和局部组件。涉及工具地图、复杂工作台、验证实验、内部模拟执行、结果看板、报告或跨页面主流程时，必须先审整个工具是否有可解释的信息架构：

- 一级页面结构：必须说明哪些页面是一等入口，例如对象列表、分析工作台、对象详情、验证工作台、仿真结果工作台、复盘中心和报告。不能把一级功能藏在某张卡片或某个报告折叠里。
- 页面职责：每个一级页面必须回答一个清楚的用户问题，而不是复用同一批卡片换标题。
- 用户路径：必须说明用户从入口、对象列表、详情证据、验证实验、内部模拟执行、复盘和报告之间如何前进与返回。
- 功能定位：必须区分“对象线索”“证据复核”“规则验证”“内部模拟操作”“报告归档”“技术审计”，不能混成一个字段看板。
- 入口与结果回看：每个一级功能都要有入口、主操作、结果看板和回看位置；验证实验和内部模拟执行不能只作为报告里的辅助字段出现。
- 字段支撑功能：每个主层字段都要能解释它服务哪个用户问题；raw enum、source path、validator status 只能作为审计支撑。
- 阶段边界：当前本地样本 / 静态 payload 阶段只是实现阶段，不得被写成最终产品定位；目标态和当前可用态要分开表达。

如果工具地图只描述现有卡片和字段，没有从用户目标出发定义一级页面、页面职责、用户问题、卡片职责和字段用途，应标为 P1/P2 IA 阻塞。

## Internal Simulation Semantics Gate / 内部模拟执行语义门槛

内部模拟执行 / 沙盒执行不是外部真实执行，不改变真实远程资源，也不能绕过守门触碰真实账号、真实资源、secret、scheduler 或 production feed。但它也不是只读审计文本。当前阶段如果只使用本地样本或静态 payload，要明确这是阶段限制，而不是最终产品定位。

涉及内部模拟执行、预演工作流、模拟操作或结果看板时，必须区分三层语义：

- 当前阶段：本地样本、静态 payload 或已验收 artifact；不得宣称真实外部执行、远程写入或生产效果证明。
- 产品内受控模拟操作：允许在受控本地状态机中表达模拟操作、模拟结果、虚拟资源或配额、指标曲线、规则阻断和复盘记录；这些是产品内部状态变化，不是外部真实执行。
- 高风险外部执行：真实账号、真实资源、secret、scheduler、production feed、自动执行、远程写入或受监管操作必须重新进入 guardian / 用户授权。

前端验收时，如果把内部模拟执行误写成外部真实执行，或反过来把内部模拟执行降格成只读说明和审计文本，都应打回。

## 前端质量标准

- 复杂工具型产品不是营销 landing page。界面应安静、密集、可扫描、可比较、适合反复使用。
- 每个页面都要回答 `interface-design` 的三件事：这个人是谁、他此刻要完成什么、界面应该让他如何判断下一步。
- 每个重要界面都要有一个来自目标项目语境的“签名元素”，例如证据链、状态边界、质量提示、业务流程脉络或结果状态，而不是通用卡片堆叠。
- 视觉选择必须能解释原因：为什么这个布局、密度、颜色、字体层级、边框/阴影策略适合金融研究工作台。
- 优先表格、筛选、状态标签、证据面板、解释面板、错误/授权状态，而不是大 hero、装饰性卡片和炫技动效。
- 必须清楚展示数据来源状态：`fixture_only`、`call_count=0`、`not_investment_advice`、provider blocked、authorization required。
- 任何评分、验证实验、内部模拟执行、风险指标或外部来源信息，都不能写成高风险外部执行指令或真实收益证明。
- 控件要有稳定尺寸，长文本不能溢出或遮挡；按钮、表格、状态徽标和导航必须在桌面和移动宽度下可读。
- 使用真实浏览器或 Playwright 截图验证用户可见路径；不能只看代码断言。
- 禁止把 `frontend-design` 的营销页倾向带入研究工作台：不要为了“好看”加入大面积 hero、装饰性渐变、低信息密度或无业务含义的动画。

## 工作流

1. 明确本次页面或组件服务哪条产品骨架链路。
2. 对照 `docs/02-design-brief.md` 查是否已有设计约束；没有则先补设计，不直接开改。
3. 检查信息架构：入口、主区域、状态区、证据区、空错态、下一步动作。
4. 检查视觉和交互：可读性、密度、响应式、键盘/鼠标路径、状态反馈、文本溢出。
5. 执行 Figma 设计审查硬门槛：首屏任务、行动路径、主次分层、审计折叠、状态语义、人话文案、窄屏可用、风险提示。
6. 执行 Human-Readable Frontend Gate：扫描主阅读层是否有 raw enum、JSON key、路径、命令、ID、调试字段或英文技术状态直露；发现即标为 P1/P2 可读性阻塞，并要求中文转译或折叠。
7. 如果涉及研究/候选/复盘/报告/行动卡片，执行 Decision-Usefulness Gate：当前判断、核心业务判断影响、关键证据、否决/风险条件、下一步人工动作、技术审计折叠和卡片类型声明。
8. 如果涉及跨页面工具或产品地图，执行 Tool-Level IA Gate：一级页面、页面职责、用户路径、入口、结果回看、状态边界和字段支撑功能。
9. 如果涉及内部模拟执行、沙盒执行或预演工作流，执行 Internal Simulation Semantics Gate：区分本地样本阶段、产品内受控模拟操作和高风险外部执行。
10. 若需要实现，交给开发泳道；若是验收，给出问题优先级和截图/命令证据。

## 打回条件

- 页面只“好看”但没有解释数据来源、授权状态、风险边界或失败状态。
- 页面字段齐全但用户看不出当前任务、下一步动作、返回路径或当前最高优先级问题。
- raw enum、审计字段、source lineage、artifact path、forbidden flags 和业务动作同层展示，导致页面像日志罗列。
- 主阅读层出现偏程序代码、偏审计日志或普通用户难以理解的信息，例如 JSON key、raw enum、内部状态名、路径、命令、thread/message id、validator 字段、DOM/API 名、脚本名或调试字段。
- 用户可见前端大面积英文或中英混杂，且没有中文解释，导致中文用户无法直接理解当前状态和下一步。
- 研究卡片、候选卡片、复盘卡片、报告卡片或行动卡片没有声明卡片类型，或主阅读层没有说明当前判断、对核心业务判断的影响、关键证据、否决/风险条件和下一步人工动作。
- 用户看完卡片仍不知道“现在能否进入下一步判断”，或只看到 `target_surface`、`source_blocker_review`、`execution_order`、`blocked_state`、`return anchors` 等机器字段。
- 质量 blocker 只作为字段出现，没有转译成“当前不支持下一步判断，因为数据质量阻断 / 证据不足 / 规则冲突 / 风险过高”等用户语义。
- 工具地图只罗列现有卡片和字段，没有定义一级页面、页面职责、用户问题、入口、结果回看和状态边界。
- 验证实验或内部模拟执行没有作为一级产品功能呈现，只藏在报告、审计详情或某张卡片里。
- 把当前本地样本 / 静态 payload 阶段误写成最终只读、不建议、不执行的产品定位。
- 把内部模拟执行误写成真实外部执行 / 真实资金 / 自动执行，或把它降格成只读审计文本，没有模拟操作、模拟结果、虚拟资源、指标曲线和复盘记录语义。
- 12 张或更多卡片同权重平铺，状态差异主要靠小标签或浅色边框，用户无法扫出风险优先级。
- 窄屏只是把桌面长审计报告纵向堆叠，没有保留当前任务摘要和主动作。
- 设计产物没有落到 `docs/02-design-brief.md`，开发泳道读不到。
- 浏览器证据缺失，却声称用户可见 UI 已完成。
- 文案暗示真实外部执行能力、真实 provider maturity 或真实收益证明。
- 大量装饰性视觉牺牲研究工作台的扫描、比较和重复使用效率。

## 完成门禁

完成时至少说明：

- 覆盖的页面/组件和用户路径。
- 更新的设计或验收文件。
- 浏览器/截图/静态 smoke 证据。
- Figma 设计审查硬门槛结论：首屏任务、行动路径、主次分层、审计折叠、状态语义、人话文案、窄屏可用和风险提示是否通过。
- Human-Readable Frontend Gate 结论：主阅读层是否中文优先，是否已把技术字段转译成人话，raw enum、JSON key、路径、命令、ID 和调试字段是否只存在于折叠审计层。
- Decision-Usefulness Gate 结论：涉及卡片时，是否声明卡片类型，是否回答当前判断、核心业务判断影响、关键证据、否决/风险条件、下一步人工动作，技术审计是否折叠。
- Tool-Level IA Gate 结论：一级页面、页面职责、用户路径、入口、结果回看、状态边界和字段支撑功能是否清楚。
- Internal Simulation Semantics Gate 结论：是否清楚区分本地样本阶段、产品内受控模拟操作和高风险外部执行。
- 仍需开发、守门或验收处理的关注点。
