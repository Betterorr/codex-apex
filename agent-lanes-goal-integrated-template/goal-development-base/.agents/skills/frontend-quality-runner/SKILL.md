# 前端质量执行器

目标：吸收 `interface-design` 和 `frontend-design` 的高价值方法，但所有产物必须接入当前项目的 GOAL 文档、Agent Lanes worklog、dashboard 和验收证据体系。

## 吸收边界

- 默认吸收 `interface-design`：适用于工具型界面、dashboard、工作台、数据产品和交互产品的信息架构、密度、导航、状态和设计系统。
- 有条件吸收 `frontend-design`：只吸收“视觉选择来自产品语境”“避免通用 AI 模板感”“排版/层级/状态 craft”等方法。
- 不默认吸收偏营销页、大 hero、装饰性动效、泛紫色渐变、低信息密度或牺牲工作效率的设计倾向。
- 不默认吸收移动端平台专属指南；目标项目确实是移动端产品时再临时参考。

## 核心契约

- 项目文档中文优先：设计说明、验收问题、截图结论和建议动作写中文；路径、命令、DOM/API 名和错误原文保留原文。
- 前端质量结论必须回写当前项目既有文档、artifacts 或泳道 workspace，不产生孤立设计稿体系。
- 如果发现 dashboard、UI 验收或截图规则存在可复用机制缺口，记录为 signal 或交给进化泳道，不在本 Skill 内顺手扩大机制改造。

## 自然语言触发词

当前端体验质量、UI 质感、页面可读性、信息架构、交互状态、前端 polish、视觉审查、Dashboard/工作台好不好用、移动端/桌面端适配、文本溢出、按钮/表格/卡片/状态设计、浏览器截图验收被提及时，优先使用本 Skill。

## 固定输入

优先读取：

- `docs/01-product-spec.md`
- `docs/02-design-brief.md`
- `docs/03-dev-plan.md`
- `docs/05-review-report.md`
- 当前前端入口、样式、组件、页面状态和本地数据/fixture 文件。
- `agent-lanes/dashboard.md`

## 输出位置

- 设计决策、信息架构、状态规则：写入 `docs/02-design-brief.md`。
- 前端验收、截图结论、问题清单：写入 `docs/05-review-report.md` 或 `agent-lanes/lanes/review/workspace/`。
- 浏览器截图、静态 smoke、可视化证据：写入 `artifacts/<slice>/`。
- 不要创建新的设计体系文档替代 `docs/02-design-brief.md`。

## 前端质量标准

- 先判断产品语境：用户是谁、此刻要完成什么、界面如何帮助他判断下一步。
- 工具型、SaaS、CRM、数据台、研究台和运营台应安静、密集、可扫描、可比较、适合反复使用；不要套营销页构图。
- 每个重要界面都要有一个来自当前产品语境的“签名元素”，例如证据链、状态解释、来源边界、审批脉络、任务阶段或业务对象结构，而不是通用卡片堆叠。
- 视觉选择必须能解释原因：为什么这个布局、密度、颜色、字体层级、边框或阴影策略适合当前产品。
- 优先表格、筛选、状态标签、证据面板、解释面板、错误/授权状态和可复跑动作，而不是装饰性卡片和炫技动效。
- 必须清楚展示数据来源、执行状态、授权状态、失败状态和未完成边界。
- 控件要有稳定尺寸；长文本不能溢出或遮挡；按钮、表格、状态徽标和导航必须在桌面和移动宽度下可读。
- 使用真实浏览器或 Playwright 截图验证用户可见路径；不能只看代码断言。

## 工作流

1. 明确本次页面或组件服务哪条产品骨架链路。
2. 对照 `docs/02-design-brief.md` 查是否已有设计约束；没有则先补设计，不直接开改。
3. 检查信息架构：入口、主区域、状态区、证据区、空错态、下一步动作。
4. 检查视觉和交互：可读性、密度、响应式、键盘/鼠标路径、状态反馈、文本溢出。
5. 若需要实现，交给开发泳道；若是验收，给出问题优先级和截图/命令证据。

## 打回条件

- 页面只“好看”但没有解释数据来源、授权状态、风险边界或失败状态。
- 设计产物没有落到 `docs/02-design-brief.md`，开发泳道读不到。
- 缺少浏览器证据，却声称用户可见 UI 已完成。
- 文案暗示未被证据支持的真实能力、真实收益、真实生产可用或真实执行能力。
- 大量装饰性视觉牺牲工具型界面的扫描、比较和重复使用效率。

## 完成门禁

完成时至少说明：

- 覆盖的页面/组件和用户路径。
- 更新的设计或验收文件。
- 浏览器/截图/静态 smoke 证据。
- 仍需开发、守门或验收处理的关注点。
