# 缺陷修复器

目标：把一个真实缺陷从“现象”推进到“复现、根因、修复、回归验证、信号沉淀”。

## 自然语言触发词

用户提到这些词或表达时，优先使用本 Skill：

- bug、报错、异常、坏了、不能用、失败、验收失败。
- 修复、fix、debug、定位问题、复现问题、根因、回归验证。
- 测试失败、构建失败、类型检查失败、页面点不动、流程断了。
- 用户反馈问题、线上问题、回归了、之前好的现在坏了。

## 输入

优先读取：

- 用户描述、截图、日志、报错或失败命令。
- `docs/00-project-state.md`
- `docs/03-dev-plan.md`
- `docs/04-goal-log.md`
- `docs/05-review-report.md`
- 相关代码和测试。

## 核心契约

不要只修表象。先确认能否复现，再定位最小根因，修复后必须做回归验证。

## 项目文档中文优先

更新 `docs/*.md`、`agent-lanes/**/*.md`、泳道 worklog、dashboard、GOAL log、review/release/report 正文或模板说明时，默认写中文。专有名词、品牌、代码、命令、路径、API 名、JSON key、状态枚举、错误原文、许可证原文和外部引用标题可以保留英文或原文。

## 系统化调试纪律

- 先复现或解释为什么无法复现，再改代码。
- 每次只验证一个主要假设；优先找最小根因，不用大范围猜修。
- 如果连续 3 次修复尝试仍失败，暂停扩大修改，重新检查输入、环境、依赖、调用路径和架构假设。
- 对路径、cwd、环境变量、编码、参数映射、输出目录、fixture 接线等常见运行问题，修复后必须复跑同一失败路径，不能只跑旁路 mock。

## 必须输出的证据

- 复现方式或无法复现的原因。
- 根因判断。
- 修改范围。
- 回归验证命令或手动验收。
- 是否需要更新产品、设计、开发计划、审查报告或 changelog。

## 打回条件

- 没有复现证据就大范围改代码。
- 没有根因假设或最小验证路径就连续试修。
- 修复后没有回归验证。
- 只解释原因但没有修复或阻塞说明。
- 把用户反馈当成一次性事件，未判断是否应记录 signal。

## 信号记录

如果 bug 暴露了可复发的流程问题，例如验收标准缺失、测试覆盖缺口、文档漂移、原型复用误判、发布记录缺失，在 `.codex/signals/` 写 signal。

## 完成门禁

只有缺陷被修复并通过回归验证，或明确记录为阻塞且给出下一步，才算完成。
