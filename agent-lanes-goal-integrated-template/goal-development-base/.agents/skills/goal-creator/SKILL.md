# GOAL 创建器

目标：写出能驱动自主执行循环的 GOAL。

## 项目文档中文优先

更新 `docs/*.md`、`agent-lanes/**/*.md`、泳道 worklog、dashboard、GOAL log、review/release/report 正文或模板说明时，默认写中文。专有名词、品牌、代码、命令、路径、API 名、JSON key、状态枚举、错误原文和外部引用标题可以保留英文或原文。

## 自然语言触发词

用户提到这些词或表达时，优先使用本 Skill：

- GOAL、目标、帮我写目标、写目标、把这个变成 GOAL、下一阶段目标、可执行目标。
- 自动执行、自主执行、一次性跑完、让 AI 自己跑、闭环。
- 完成标准、验收方式、验证方式、循环规则、失败了继续修。
- 执行这个计划、把 phase 写成目标、不要我一步步盯。
- 多泳道派发目标、泳道任务格式。

## 输入

优先读取：

- `docs/00-project-state.md`
- `docs/03-dev-plan.md`
- `docs/04-goal-log.md`
- `.codex/goals/goal-template.md`
- 当前对话中的用户请求。

## 核心契约

GOAL 的价值是定义结束条件。不要只写目标，要写清完成标准、验证证据和失败后的循环规则。

## 多泳道下的按需使用

启用 Agent Lanes 后，`goal-creator` 不再是每次派发前的必经站。主调度可以直接发送结构化泳道任务，只要任务已经包含目标、范围、完成标准、验证方式、禁止事项、证据格式和 completion callback 要求。

仍然需要使用本 Skill 的情况：

- 用户请求或下一步目标模糊，无法判断完成/失败。
- 任务跨多个泳道且依赖关系、验收标准或禁止事项不清楚。
- 中高风险、阶段边界、发布/交付、真实写入、权限/secret/外部 API 或客户可用声明。
- 之前的泳道派发因为目标、范围或验证方式不清导致返工。

不需要单独使用本 Skill 的情况：

- 主调度已经能用短结构化消息清楚派发低风险单泳道任务。
- 任务只是补文档、跑门禁、补证据、局部小修或复核 callback。
- 当前最缺的是执行或验收，而不是重新写一个正式 GOAL。

## GOAL 格式

```text
GOAL:
[具体结果]

范围:
- 包含:
- 不包含:

完成标准:
- [可验证标准]
- [可验证标准]
- [可验证标准]

验证方式:
- [命令、浏览器流程、截图、日志、审查报告或交付物]

自动门禁:
- [需要 gate-runner 运行或创建的构建、测试、类型检查、文档、审查或发布门禁]

文档更新:
- [必须同步更新的文档]

子 Agent / 独立审查:
- [是否需要 explorer、worker、reviewer，以及派发上下文]

循环规则:
- 如果任一标准失败，修复后重新验证。
- 所有必要证据存在之前，不报告完成。
```

## 打回条件

- 目标无法判断完成或失败。
- 完成标准不能被命令、截图、日志、测试或审查证明。
- 范围边界缺失。
- 没有说明文档更新。
- 能用程序判断的完成条件没有写入自动门禁。
- 高风险任务没有独立审查要求。

## 子 Agent 安排

写 GOAL 时必须显式判断是否需要子 Agent / 新脑子：

- 涉及代码实现：要求完成后由 `code-reviewer` 做未参与实现的独立代码审查。
- 涉及用户可见流程、跨模块行为或验收闭环：要求完成前由 `review-runner` 核对证据和文档。
- 涉及安全、隐私、权限、数据、发布或生产风险：要求独立安全/发布审查。
- 存在可并行且互不重叠的任务：可以规划 worker 子 Agent，但必须写清文件/模块边界。

GOAL 里的子 Agent 段必须写清：角色、触发时机、要带走的上下文、返回证据格式。

## 输出

返回一条可直接发送执行的 GOAL。若用户已经要求执行，先写出 GOAL，再进入执行。

## 信号记录

当 GOAL 因范围、验收或验证不足导致跑偏时，在 `.codex/signals/` 记录 signal。

## 完成门禁

只有当 GOAL 能让 `dev-builder` 自主进入“执行、验证、修复、复审、文档更新”循环时，GOAL 才算合格。
