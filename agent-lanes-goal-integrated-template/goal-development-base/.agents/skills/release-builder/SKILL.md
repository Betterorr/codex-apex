# 发布构建器

目标：让发布准备状态变成清晰、可验证、可回滚的记录。

## 自然语言触发词

用户提到这些词或表达时，优先使用本 Skill：

- 发布、上线、部署、打包、交付、release、deploy。
- 生产就绪、试运行、发布记录、交接、客户交付。
- 环境变量、启动方式、日志、回滚、冒烟测试。
- 隐私审计、安全审计、上线前检查、配置项、部署地址、运行记录。
- 模型依赖、外部 API 依赖、Provider 依赖、依赖剥离。

## 输入

优先读取：

- `docs/00-project-state.md`
- `docs/03-dev-plan.md`
- `docs/04-goal-log.md`
- `docs/05-review-report.md`
- 当前部署、打包或运行配置。

## 核心契约

发布不是“代码能跑”。必须能说明怎么启动、怎么验证、怎么看日志、怎么回滚、哪些配置必需、还有什么风险。

## 项目文档中文优先

更新 `docs/*.md`、`agent-lanes/**/*.md`、泳道 worklog、dashboard、GOAL log、review/release/report 正文或模板说明时，默认写中文。专有名词、品牌、代码、命令、路径、API 名、JSON key、状态枚举、错误原文、许可证原文和外部引用标题可以保留英文或原文。

## 子 Agent 安排

发布前如果涉及安全、隐私、权限、数据、外部服务、生产环境或客户交付，必须安排独立新脑子审查：

- 代码风险交给 `code-reviewer`。
- GOAL 闭环、发布证据、回滚和日志交给 `review-runner`。

派发时必须带上：发布目标、配置项名称、启动方式、日志方式、回滚方式、冒烟测试证据、已知限制和禁止泄露 secret 的要求。

## 发布记录必须包含

- 版本号或发布标识。
- 目标环境或交付物。
- 必需配置项名称，不暴露 secret 值。
- 启动方式。
- 日志查看方式。
- 回滚方式。
- 冒烟测试证据。
- 隐私和安全说明。
- 已知限制。
- 涉及外部 API、云端 Provider、本地模型、CLI、SDK、媒体处理工具或其他真实运行依赖时，必须列出依赖安装/下载方式、配置项名称、是否打包进主程序、是否作为外部依赖、真实 smoke/sample evidence、复用 artifact 位置、调用次数/预算边界和 secret 不落盘说明。

## 自动门禁

发布相关 GOAL 必须使用 `gate-runner` 核对：

- 发布记录是否完整。
- 启动方式是否可执行。
- 日志查看方式是否明确。
- 回滚方式是否明确。
- 冒烟测试是否有证据。
- 必需配置项是否列出且未泄露 secret。
- 隐私和安全说明是否完成。
- 模型/API/Provider 依赖是否能从主程序包中剥离，且有 README、CLI、配置和验证说明供新环境重新下载、配置和验证。

## 打回条件

- 没有启动方法。
- 没有日志查看方法。
- 没有回滚方法。
- 没有冒烟测试证据。
- 模型/API/Provider 依赖、CLI、README、真实样本 evidence 或剥离打包边界缺失。
- 有阻塞审查问题仍宣称可发布。
- `gate-runner` 发布门禁失败。
- secret 值被写进文档。

## 输出

更新：

- `docs/06-release-record.md`
- `docs/CHANGELOG.md`
- 必要时更新 `docs/00-project-state.md`

## 信号记录

当发布失败暴露环境变量、部署地址、日志、回滚、隐私或验收记录缺失时，在 `.codex/signals/` 记录 signal。

## 完成门禁

启动方式、冒烟测试、日志查看、回滚方式和阻塞审查状态都明确后，才能宣称发布准备完成。
